Relational db anyone?

* when you first learned about databases, you were probably told that this is how a db looks like * tables, columns (properties), rows * highly optimized, very mature, great tooling, everyone knows SQL * Most popular model for a reason, please use it 98% of the time * they have some fundamental downsides, interestingly esp. when it comes to managing relations * have a look at that schema: person, pizza, likes (many to many): what pizzas does nicola like?

relations in sql


SELECT pizza.name
FROM Person as person
JOIN Likes as likes ON person.id = likes.person
JOIN Pizza as pizza ON likes.pizza = pizza.id AND
WHERE person.id = 1
vs. gremlin 

              g.V(1).out("likes").values("name")
* don't focus on the details here - what I wanted to show is that for such a simple task we need a lot of ceremony * also gets slow when following multiple relations: * global index: lookup time dependent on index size: O(log(n)) complexity - two in this instance * node-centric index: O(1) constant time * imagine how it looks when we're asking for pizza recommendations: which pizzas do people like, who also like the pizzas alice likes? * constant juggle: normalization (clean schema) vs. performance (wide rows) * object-relational impedance mismatch: if a value class has a `list` property, you can either store that list in a separate table, or encode it in some form (json, proto, ...)

Property Graph

* quick definition: when I say "graph" I actually mean "property graph" * vertices connected by directed edges, each of which have one label and 0..n properties * simple, natural model * schema typically optional: no schema for exploratory stuff, define schema once domain is better understood Consider graphs for: * their simpler model, no fight with object-relational impedance mismatch * highly relational domains

(friends of friends)^n

* archetypical motivational example: social network, finding friends of friends * 1M people, each with ~50 friends * img: attention, propaganda. from the free oreilly graphdb book (written by neo4j). * you can optimise your data in a relational db to workaround that, but you get this for free in a graphdb
* example, to get an idea where a graph makes complete sense * code property graph * parse source code (java, .net, js, ...) into this language neutral form, which is our domain * control flow, data flow, program dependence * run traversals to find meaning and find things that are connected, that maybe shouldn't be connected. * password coming in via http that ends up in a logfile (hello twitter) * sql, bytecode, filepath injections * traverse many connections

Query languages

* query/traversal * given a graph with vertices and edges, there's different options to retrieve information (query/traverse it)

Cypher


MATCH (Actor {name: 'Woody Allen'})
      -[:ACTED_IN]->(movie:Movie)
WHERE movie.year > 2010
RETURN movie
* get woody allen movies past 2010 * declarative, similar to how you would draw it * probably great for many use cases, also hear of problems with it, not an expert * originally created by Neo4j -> openCypher: SAP HANA, Redis, gremlin, ... * libs: [cypher for spark](https://github.com/opencypher/cypher-for-apache-spark), [AnormCypher](https://github.com/AnormCypher/AnormCypher) * but you still write the cypher query as a string, i.e. no type safety, mostly helps with scaffolding

SPARQL (W3C)


PREFIX info:  <http://somewhere/peopleInfo#>
PREFIX vcard: <http://www.w3.org/2001/vcard-rdf/3.0#>

SELECT ?name ?age
WHERE
{
    ?person vcard:FN  ?name .
    OPTIONAL { ?person info:age ?age . }
    FILTER ( !bound(?age) || ?age > 24 )
}
* get names of people older than 24 years * for RDF (Resource Description Framework): making statements about resources, in expressions of the form subject–predicate–object, known as triples. * low level primitive, one can build additional ontology languages upon RDF * format naming and definition of types/properties/relationships of entities in a domain * found only one scala lib which was for specific spark, against create your own strings

Gremlin (Apache TinkerPop)


g.V().has("name","gremlin")
     .out("friend")
     .out("friend")
     .values("name")
* What are the names of Gremlin's friends' friends? * main style: imperative, declarative style available (match step) * steps: combine and pipe together * turing complete / computationally universal * implemented by a long list of DBs * libraries for many host languages: java, groovy, python, js, .net

https://github.com/mpollmeier/gremlin-scala
* scala wrapper for gremlin-java * scala friendly function syntax * objective: use scala's advanced type system * help in your IDE by deriving types * prevent families of illegal queries * make use of the fact that scala is great for embedding DSLs

setup


              import gremlin.scala._
              implicit val graph = TinkerGraph.open.asScala
* reference to graph - could be local (in memory, backed by fs), remote

schema

define once, use everywhere 

object PersonNode {
  val name = Key[String]("name")
  val age = Key[Int]("age")
}

object LikesEdge {
  val howMuch = Key[Double]("howMuch")
}
* graph db may or may not enforce the same schema * define domain schema once, use everywhere to get help from compiler/IDE and type safety and IDE completion * we defined our schema in json, used for all sorts of things: protos, domain classes, other language impl

arrow syntax


              val alice = graph + ("person", name -> "alice", age -> 35)
              val tonno = graph + ("pizza", name -> "tonno")

              alice --- "likes" --> tonno
              alice --- ("likes", howMuch -> 3) --> tonno
* properties: type safe: IDE help / compiler validation * edges can be the other way around, or bidirectional * can use case classes for adding and retrieving vertices, as well * user contribution * made me realise how nice scala is for embedding a dsl

case classes


              @label("person")
              case class Person(@id id: Option[Int] = None,
                                name: String,
                                age: Option[Int])

              val alice = graph + Person(name = "alice", age = Some(35))
              alice.toCC[Person] // Person with id set
* (blackbox) macro for mapping vertices from/to case classes * handles Option[A]: automatically unwraps it * id is set by the graph * has some rough edges (and a few open tickets), e.g. around using list * was about to rewrite in scalamacros backed by scala.meta, but it felt like progress has stalled and direction a little unclear

traversal basics


              val g = graph.traversal
            

              g.V // GremlinScala[Vertex]
              g.V.toList // List[Vertex]
            

              g.V(1) // GremlinScala[Vertex]
              g.V(1).headOption // Option[Vertex]
* `g` convention for naming the TraversalSource * lazy: execute with `toList`, `toSet`, `headOption` etc.

out step


g.V(1).out("likes") // GremlinScala[Vertex]
            

g.V(1).outE("likes").inV // GremlinScala[Vertex]

has step

g.V.hasLabel("person") // GremlinScala[Vertex]
g.V.has(name) // GremlinScala[String]
g.V.has(age, P.between(25, 35)) // GremlinScala[Int]
* IDE helps with types * P: Predicate: only compiles with two Ints

group


                g.V.group(By.label)
                // GremlinScala[Map[String, Collection[Vertex]]]
            

                g.V.group(By(__.outE.count))
                // GremlinScala[Map[Long, Collection[Vertex]]]
            

                g.V.has(age).groupCount(By(age))
                // GremlinScala[Map[Int, Long]]
* degree: how many outgoing edges from a specific vertex

repeat


                g.V(1)
                .repeat(_.out)
                .until(_.outE.count.is(P.eq(0)))
* traverse from one starting node until the outermost vertices in the graph, which don't have any outgoing edges any more: discover the new frontier

as/select


              g.V.as("a")
                .outE.as("b")
                .value(howMuch).as("c")
                .select
              // GremlinScala[(Vertex, Edge, Double)]
* selecting multiple things along the way * groovy/java would return Array[Object] (at compile time) * also possible to select just a subset of the labelled steps * works by storing the labelled types in a type member as an HList

catch errors at compile time


              g.V.outE.outE
            

              g.V(1).property(name, 5) //name: Key[String]
            

              g.V.value(name).mean
* prevent families of illegal queries * dynamic languages can't catch these * simpler compilers like java can only detect a subset

Downsides: Tinkerpop

  • db maturity
  • db driver
  • tooling
* graph db maturity - rapidly expanding space, however mixed maturity * quality of db driver: query optimisation, automatic use of indices * tooling: far behind the steam of relational dbs * prepare to step in and help in the spirit of libre & open source software, ask your employer to not just take but also contribute back. * gremlin-scala: plenty of areas for improvement, help appreciated, and already got many great contributions

Summary

  • graphs
  • gremlin-scala: typesafe dsl
  • nice DSL to create vertices and edges
* graphs: simpler model, better performance for connected data * optional schema: faster turnaround in early phases, while exploring domain * gremlin: api for local and remote graphs * Minimal runtime overhead - only allocates additional instances if absolutely necessary * Nothing is hidden away, you can always easily access the underlying Gremlin-Java objects if needed, e.g. to access graph db specifics things like indexes
